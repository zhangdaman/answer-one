## 二、数据契约摘要

> 完整数据契约见 `docs/PRD.md` §9；接口契约见 `docs/api-contracts.md`。

### 统一响应格式
- 成功：`{"code": 200, "message": "success", "data": { ... }}`
- 错误：`{"code": <错误码>, "message": "<错误描述>", "data": null}`
- 分页：`{"code": 200, "data": {"items": [...], "total": N, "page": N, "page_size": N}}`
- 异步诊断：发起 `{"code": 200, "data": {"task_id": "...", "diagnosis_id": "...", "status": "running"}}`；进度 `{"code": 200, "data": {"status": "running", "progress": "3/10", ...}}`

### 核心实体
`User 1:1 Brand`；`Brand 1:N Diagnosis / Content / Routing`；`Diagnosis 1:N Question / Mention`。枚举值（含中英映射）以 api-contracts「枚举字典」为准，后端用字典表维护。

### 总分算法（V1）
`score_total = 提及率分×0.5 + 位置分×0.3 + 情感分×0.2`，三项均归一到 0–100。

### 后端任务验收规则
- **基础设施基于 pycore 框架**：禁止自己重写 config.py / server.py / logger.py；配置用 `pycore.core.ConfigManager`，服务器用 `pycore.api.APIServer`，数据库骨架从 `pycore/integrations/db/` 复制模板。
- 基础设施任务由 Agent 自动连续执行，不触发用户门禁；验收：ruff/mypy 通过、单元测试通过、`GET /health` 返回 200。
- 对应前端页面/Service 的业务任务必须**任务内真实联调**（后端 API + 前端 `VITE_USE_MOCK=false` 切真实），每个功能是完整闭环。
- 真实联调验收至少包含：前端调真实后端、核心操作可用、页面无 `[Mock]` 数据/账号提示、Tester 能证明请求命中真实后端而非 `frontend/src/mocks/*`。
- 涉及外部服务的任务必须引用「外部服务与测试权限清单」；Key/权限缺失只能标记 Mock/fallback 降级验收，不得宣称真实外部服务联调通过。
- 最终 E2E / 回归只做全系统复查，不承担第一次前后端联调。

## 六、开发顺序建议

> 默认推荐节奏，实际以 Planner 产出的 `.sdd/tasks.json` 为准。

**阶段 1：前端 MVP（Mock，用户先验收 UI/UX）**
1. P00 壳与字典 → P01 登录 → P02 建档 → P03 首页 → P04 档案 → P05 诊断列表 → P06 报告 → P07 生成 → P08 内容库 → P09 分发
2. 验收（用户门禁）：布局、配色、交互、响应式与原型一致

**阶段 2：后端基础设施（自动连续执行，不触发用户门禁）**
1. 基于 pycore 初始化结构、配置、数据库、Celery、中间件
2. 验收（Agent 自动）：ruff/mypy 通过、单元测试通过、`GET /health` 返回 200

**阶段 3：逐功能闭环开发（每个功能完成后触发用户门禁）**
1. 顺序：认证 → 品牌档案 → 诊断引擎 → 诊断报告/PDF → 内容生成/内容库 → 分发建议/导出 → 工作台聚合
2. 每个功能 = 后端真实 API + 前端切真实 + 联调验收
3. 验收（用户门禁）：用户从前端操作验证可用

**阶段 4：E2E 回归测试**
1. 黄金路径走通：登录 → 建档 → 诊断 → 报告 → 生成 → 内容库 → 分发 → 标记已发布 → 复诊对比
2. 验收（用户门禁）：全流程通过，无阻塞性 Bug
